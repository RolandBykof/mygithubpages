Käytettävät palvelut ja työkalut

iOS-sovelluksen julkaisu edellyttää useiden palveluiden ja työkalujen käyttöä. Ne täydentävät toisiaan ja jokaisella on oma tehtävänsä prosessissa.

🔐

Apple Developer Portal

developer.apple.com

Applen kehittäjäportaali, jossa hallinnoidaan kaikkia teknisiä tunnistetietoja ja oikeuksia iOS-kehitykseen.

  • Bundle ID:n rekisteröinti sovellukselle
  • Kehittäjäsertifikaattien hallinta
  • Provisioning profiilien luonti ja hallinta
  • Testattavien laitteiden rekisteröinti
Vaatii Developer Program -jäsenyyden
🛍

App Store Connect

appstoreconnect.apple.com

Applen hallintaportaali julkaistuille ja julkaistaville sovelluksille. Täällä sovellus luodaan, hallitaan ja lähetetään arviointiin.

  • Sovelluksen luonti ja metatietojen hallinta
  • TestFlight-jakelun hallinta
  • Buildien vastaanotto ja seuranta
  • API-avainten luonti CI/CD-integraatioita varten
  • Sovelluksen lähettäminen App Store -arviointiin
Sama Apple ID kuin Developer Portal
⚙️

Codemagic

codemagic.io

CI/CD-palvelu (Continuous Integration / Continuous Delivery) joka rakentaa iOS-sovelluksen automaattisesti pilvessä. Mahdollistaa iOS-buildin ilman Mac-konetta – Codemagic ajaa buildin omalla Mac-koneellaan.

  • Hakee koodin automaattisesti GitHubista
  • Rakentaa IPA-tiedoston (iOS-asennuspaketti)
  • Allekirjoittaa sovelluksen sertifikaatilla
  • Lähettää valmiin buildin App Store Connectiin
  • Tallentaa sertifikaatit ja provisioning profiilit
Ilmainen taso saatavilla Yhdistetään GitHub-tiliin
🐙

GitHub

github.com

Versionhallintapalvelu, jossa sovelluksen lähdekoodi säilytetään. Codemagic hakee koodin automaattisesti GitHubista aina kun uusi build käynnistetään.

  • Lähdekoodin säilytys ja versionhallinta
  • Codemagic yhdistetään repositorioon
  • codemagic.yaml sijaitsee repon juuressa
  • Jokainen commit voidaan ajaa automaattisesti buildina
Ilmainen julkisille ja yksityisille repoille
💙

Flutter

flutter.dev

Googlen kehittämä ohjelmistokehys mobiilisovellusten rakentamiseen. Yhdellä koodilla voi rakentaa sovelluksen sekä iOS:lle että Androidille. Käyttää Dart-ohjelmointikieltä.

  • Sovelluksen lähdekoodi kirjoitetaan Dartilla
  • Asennetaan kehittäjän omalle koneelle
  • Komennot ajetaan komentoriviltä
  • Codemagic käyttää Flutteria pilvessä buildin aikana
Ilmainen ja avoimen lähdekoodin

Palveluiden välinen suhde

Prosessin kulku palveluiden välillä on seuraava:

  1. GitHub toimii lähdekoodin säilytyspaikkana. codemagic.yaml-tiedosto repon juuressa ohjaa buildin kulkua.
  2. developer.apple.com rekisteröi sovelluksen Bundle ID:n ja antaa oikeuden allekirjoittaa sovelluksia Distribution-sertifikaatilla.
  3. App Store Connect vastaanottaa valmiin buildin ja hallinnoi TestFlight-jakelua sekä mahdollista App Store -julkaisua.
  4. Codemagic toimii välittäjänä: hakee koodin GitHubista, rakentaa ja allekirjoittaa sovelluksen, ja lähettää sen App Store Connectiin automaattisesti.
ℹ
API-avain yhdistää palvelut App Store Connect API -avain on keskeinen linkki Codemagicin ja Applen palveluiden välillä. Se antaa Codemagicille luvan luoda sertifikaatteja, hakea provisioning profiileja ja lähettää buildeja – ilman että sinun tarvitsee tehdä näitä manuaalisesti joka kerta.
FL

Flutter – asennus ja peruskäyttö

Flutter on Googlen kehittämä avoimen lähdekoodin ohjelmistokehys, jolla voi rakentaa mobiili-, web- ja työpöytäsovelluksia yhdellä koodilla. Ohjelmointi tehdään Dart-kielellä. Flutter asennetaan kehittäjän omalle koneelle, ja samaa Flutteria käyttää myös Codemagic pilvessä buildin aikana.

Asennus Windowsille

  1. Mene osoitteeseen flutter.dev/docs/get-started/install/windows ja lataa Flutter SDK -paketti (.zip).
  2. Pura zip-tiedosto hakemistoon johon sinulla on kirjoitusoikeus, esim. C:\flutter. Älä pura C:\Program Files-kansioon.
  3. Lisää Flutterin bin-hakemisto PATH-ympäristömuuttujaan:
    C:\flutter\bin Tämä tehdään: Ohjauspaneeli → Järjestelmä → Lisäasetukset → Ympäristömuuttujat → Path → Muokkaa → Uusi
  4. Avaa uusi komentokehote ja tarkista asennus: 
    flutter --version
  5. Tarkista puuttuvat riippuvuudet: 
    flutter doctor
    Komento kertoo mitä vielä tarvitaan. iOS-kehitykseen ei tarvita kaikkia kohtia – Codemagic hoitaa iOS-buildin.
ℹ
iOS-buildi onnistuu ilman Xcodea paikallisesti Windows-koneella ei voi buildata iOS-sovellusta suoraan, koska se vaatii macOS:n ja Xcoden. Codemagic hoitaa iOS-buildin pilvessä omalla Mac-koneellaan – sinun ei tarvitse asentaa Xcodea.

Uuden projektin luominen

Luo uusi Flutter-projekti komentoriviltä:

flutter create sovelluksen_nimi

Komento luo projektin oletuksena kaikille alustoille (iOS, Android, Web jne.). Jos haluat rajata projektin vain iOS:lle:

flutter create --platforms ios sovelluksen_nimi

Jos haluat määrittää samalla organisaation ja bundle ID:n:

flutter create --org fi.villelamminen --platforms ios sovelluksen_nimi

Tällöin bundle ID:ksi tulee automaattisesti fi.villelamminen.sovelluksen_nimi. Voit lisätä muita alustoja myöhemmin komennolla:

flutter create --platforms android .

Tämä luo hakemistorakenteen automaattisesti. Tärkeimmät tiedostot:

  • lib/main.dart – sovelluksen pääkoodi, tänne kirjoitetaan sovelluksen logiikka.
  • pubspec.yaml – projektin konfiguraatio: nimi, versio ja riippuvuudet (käytettävät kirjastot).
  • ios/ – iOS-kohtaiset tiedostot, mukaan lukien bundle ID ja Xcode-projekti.
  • android/ – Android-kohtaiset tiedostot.
  • codemagic.yaml – Codemagic-konfiguraatio, luodaan projektin juureen manuaalisesti.

Riippuvuuksien asentaminen

Flutterin kirjastot (paketit) määritellään pubspec.yaml-tiedostossa dependencies-lohkossa. Kun olet lisännyt paketin, asenna se komennolla:

flutter pub get

Komento lataa kaikki määritellyt paketit verkosta ja tekee ne käytettäviksi projektissa. Aja tämä aina kun:

  • Kloonaat projektin GitHubista uudelle koneelle.
  • Lisäät tai muutat paketteja pubspec.yaml-tiedostossa.
  • Projekti antaa virheen "package not found".

Koodin tarkistus ennen buildia

Ennen kuin ajat Codemagicissa kalliin buildin, kannattaa tarkistaa koodi paikallisesti. Tämä säästää aikaa ja Codemagicin build-minuutteja.

Staattinen analyysi – löytää tyyppivirheet ja syntaksivirheet:

flutter analyze

Tallenna tulos tiedostoon myöhempää tarkastelua varten:

flutter analyze > tulos.txt 2>&1

Tarkista riippuvuuksien päivitystarpeet:

flutter pub outdated

Päivitä riippuvuudet uusimpiin yhteensopiviin versioihin:

flutter pub upgrade
⚠
Yleisiä analyze-virhejä Jos analyze ilmoittaa "Cannot invoke a non-'const' constructor where a const expression is expected", muuta const → final virheen osoittamalla rivillä. Jos ilmoittaa "dot-shorthands language feature", kirjoita lyhytsyntaksi auki: .center → MainAxisAlignment.center.

Muita hyödyllisiä komentoja

# Näytä Flutter-version tiedot
flutter --version

# Tarkista kehitysympäristön tila
flutter doctor

# Listaa asennetut paketit
flutter pub deps

# Siivoa buildin väliaikaistiedostot
flutter clean

# Aja sovellus liitetyllä laitteella tai emulaattorilla
flutter run

# Listaa käytettävissä olevat laitteet
flutter devices

CocoaPods ja Swift Package Manager

Mikä on CocoaPods?

CocoaPods on ollut vuosien ajan iOS-kehityksen tärkein riippuvuudenhallintajärjestelmä. Flutter on käyttänyt sitä iOS-lisäosien (pluginien) hallintaan. Flutter luo projektin luonnin yhteydessä automaattisesti ios/Podfile-tiedoston, joka määrittelee käytettävät CocoaPods-riippuvuudet. Codemagicin YAML:ssa oleva pod install -vaihe asentaa nämä riippuvuudet buildin aikana.

⚠
CocoaPods poistuu käytöstä CocoaPods siirtyy vain ylläpitotilaan 2. joulukuuta 2026. Tämän jälkeen uusia paketteja ei voi lisätä eikä olemassa olevia päivittää. Flutter on siirtymässä Swift Package Manageriin (SPM), joka on Applen natiivi vaihtoehto. Siirtyminen on suositeltavaa tehdä ennen määräaikaa.

Mikä on Swift Package Manager?

Swift Package Manager (SPM) on Applen oma riippuvuudenhallintajärjestelmä, joka on sisäänrakennettu Xcodeen. Se ei vaadi Rubya tai CocoaPodsin asennusta. Flutter 3.44:stä alkaen SPM on oletusarvoinen riippuvuudenhallintajärjestelmä kaikille Flutter-sovelluksille, jotka kohdistuvat Apple-alustoille.

Siirtyminen CocoaPodsista Swift Package Manageriin

ℹ
Ennen siirtymistä Siirtyminen vaatii Flutter 3.24 tai uudemman. Tarkista versio komennolla flutter --version. Lisäksi kaikkien käyttämiesi pluginien tulee tukea SPM:ää – tarkista tämä pluginin GitHub-repositoriosta (Package.swift-tiedoston olemassaolo kertoo SPM-tuesta).
  1. Päivitä Flutter uusimpaan versioon: 
    flutter upgrade
  2. Tarkista että kaikki pluginit tukevat SPM:ää: 
    flutter pub deps
    Käy jokaisen pluginin GitHub-sivulla ja tarkista löytyykö Package.swift-tiedosto.
  3. Poista CocoaPods-integraatio projektin ios/-hakemistosta: 
    cd ios
pod deintegrate
    Tämä komento vaatii CocoaPodsin asennuksen koneella. Windows-käyttäjät voivat poistaa seuraavat tiedostot manuaalisesti: Podfile, Podfile.lock ja Pods/-kansio.
  4. Ota Swift Package Manager käyttöön projektissa lisäämällä pubspec.yaml-tiedostoon: 
    flutter:
  ios:
    swift_package_manager:
      enabled: true
  5. Päivitä riippuvuudet: 
    flutter pub get
    Flutter luo automaattisesti tarvittavat SPM-tiedostot.
  6. Päivitä Codemagicin YAML poistamalla pod install -vaihe kokonaan – sitä ei enää tarvita: 
    # Poista tai kommentoi tämä vaihe:
# - name: Install CocoaPods
#   script: |
#     find . -name "Podfile" -execdir pod install \;
✓
Onnistuneen siirtymisen merkit Buildilokeissa ei enää näy varoitusta "All plugins found for ios are Swift Packages, but your project still has CocoaPods integration". Build on myös nopeampi koska Ruby- ja CocoaPods-vaihe jää pois.

Edellytykset

Ennen kuin aloitat, varmista että sinulla on seuraavat:

  • Aktiivinen Apple Developer Program -jäsenyys (99 $/vuosi). Tilisi roolin tulee olla Account Holder.
  • Flutter-projekti GitHub-repositoriossa.
  • Tili codemagic.io-palvelussa, joka on yhdistetty GitHub-tiliisi.
  • Sovelluksen pubspec.yaml-tiedostossa on asetettu version-kenttä, esim. version: 1.0.0+1.
⚠
Huomio Flutter-versiosta Flutter 3.29+ käyttää Swift Package Manageria. Jos koodissasi on dot-shorthands-syntaksia (esim. .center tai .fromSeed()), käytä täydellistä syntaksia (MainAxisAlignment.center, ColorScheme.fromSeed()) jotta build onnistuu.
02

Bundle ID:n luominen Apple Developer Portalissa

Bundle ID on sovelluksesi yksilöllinen tunniste, kuten fi.yrityksesi.sovellus. Se luodaan selaimen kautta eikä vaadi Mac-konetta.

developer.apple.com→ Account→ Certificates, IDs & Profiles→ Identifiers→ +
  1. Valitse tyypiksi App IDs ja klikkaa Continue.
  2. Valitse tyypiksi App ja klikkaa Continue.
  3. Kirjoita Description-kenttään sovelluksesi nimi.
  4. Valitse Explicit Bundle ID -tyypiksi ja kirjoita haluamasi Bundle ID käänteisessä verkkotunnusmuodossa, esim. fi.villelamminen.matkamittari.
  5. Klikkaa Continue ja sitten Register.
ℹ
Bundle ID:tä ei voi muuttaa rekisteröinnin jälkeen. Varmista että se vastaa täsmälleen Flutter-projektin ios/Runner.xcodeproj-tiedostossa määriteltyä Bundle Identifier -arvoa.
03

Sovelluksen luominen App Store Connectissa

Sovellus täytyy luoda App Store Connectiin ennen kuin Codemagic voi lähettää buildeja sinne.

appstoreconnect.apple.com→ Apps→ +→ New App
  1. Platforms: valitse iOS.
  2. Name: sovelluksesi nimi julkisessa App Storessa.
  3. Primary Language: sovelluksen pääkieli.
  4. Bundle ID: valitse aiemmin luomasi Bundle ID pudotusvalikosta.
  5. SKU: keksi jokin yksilöllinen tunniste, esim. matkamittari2024. Ei näy käyttäjille.
  6. Klikkaa Create.
04

Apple API -avaimen luominen ja lisääminen Codemagiciin

Luo avain App Store Connectissa

appstoreconnect.apple.com→ Users and Access→ Integrations→ App Store Connect API→ +
  1. Anna avaimelle nimi, esim. Codemagic Admin.
  2. Valitse rooliksi Admin.
  3. Klikkaa Generate.
  4. Lataa .p8-tiedosto välittömästi – sitä ei voi ladata uudelleen myöhemmin.
  5. Kopioi talteen Issuer ID ja Key ID.

Lisää avain Codemagiciin

codemagic.io→ Settings→ Integrations→ Developer Portal→ Manage keys → Add key
  1. Anna avaimelle nimi, esim. codemagic_admin. Tätä nimeä käytetään codemagic.yaml-tiedostossa.
  2. Syötä Issuer ID ja Key ID.
  3. Lataa .p8-tiedosto.
  4. Klikkaa Save.
05

Sertifikaatin luominen Codemagicissa

iOS-sovellukset täytyy allekirjoittaa digitaalisella sertifikaatilla. Codemagic voi luoda sen suoraan – Mac-konetta ei tarvita.

codemagic.io→ Settings→ codemagic.yaml settings→ Code signing identities→ iOS certificates→ Generate certificate
  1. Anna sertifikaatille Reference name, esim. Matkamittari Distribution.
  2. Valitse tyypiksi Apple Distribution.
  3. Valitse App Store Connect API -avain jonka lisäsit edellisessä vaiheessa.
  4. Klikkaa Create certificate.
  5. Lataa .p12-tiedosto ja tallenna salasana turvalliseen paikkaan.
⚠
Tallenna .p12-tiedosto ja salasana Tiedoston voi ladata vain kerran. Jos tarvitset sertifikaattia myöhemmin toisessa palvelussa, tarvitset nämä tiedot.
06

Provisioning profilen luominen ja lisääminen

Provisioning profile yhdistää sertifikaatin ja Bundle ID:n. Se luodaan selaimessa Apple Developer Portalissa.

Luo profiili Apple Developer Portalissa

developer.apple.com→ Certificates, IDs & Profiles→ Profiles→ +
  1. Valitse App Store Connect Distribution-osiosta.
  2. Valitse App ID: sovelluksesi Bundle ID.
  3. Valitse sertifikaatti: Codemagicin luoma Distribution-sertifikaatti.
  4. Anna profiilille nimi, esim. Matkamittari App Store.
  5. Klikkaa Generate ja lataa .mobileprovision-tiedosto.

Lisää profiili Codemagiciin

codemagic.io→ Settings→ codemagic.yaml settings→ Code signing identities→ iOS provisioning profiles→ Upload profile
  1. Lataa .mobileprovision-tiedosto.
  2. Anna sille Reference name, esim. Matkamittari.mobileprovision.
  3. Klikkaa Add profile.
07

codemagic.yaml -tiedoston rakenne

Luo tai korvaa repositoriosi juuressa oleva codemagic.yaml-tiedosto alla olevalla sisällöllä. Muuta vain korostettuja kohtia.

# codemagic.yaml – Flutter iOS → TestFlight
workflows:
  ios-workflow:
    name: iOS Workflow
    instance_type: mac_mini_m2

    # Vaihda arvo Codemagiciin lisäämäsi avaimen nimeksi
    integrations:
      app_store_connect: codemagic_admin

    environment:
      flutter: stable
      xcode: latest
      ios_signing:
        # Reference namet Code signing identities -osiosta
        provisioning_profiles:
          - Matkamittari.mobileprovision
        certificates:
          - Matkamittari Distribution

    scripts:
      - name: Flutter pub get
        script: |
          flutter pub get

      - name: Get Flutter packages
        script: |
          flutter packages pub get

      - name: Install CocoaPods
        script: |
          find . -name "Podfile" -execdir pod install \;

      - name: Set up code signing
        script: |
          xcode-project use-profiles

      - name: Create export options plist
        script: |
          cat > /tmp/ExportOptions.plist << PLIST
          <?xml version="1.0" encoding="UTF-8"?>
          <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
            "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
          <plist version="1.0">
          <dict>
            <key>method</key>
            <string>app-store</string>
            <key>signingStyle</key>
            <string>manual</string>
            <key>provisioningProfiles</key>
            <dict>
              <key>fi.villelamminen.matkamittari</key>
              <string>Matkamittari</string>
            </dict>
          </dict>
          </plist>
          PLIST

      - name: Build IPA
        script: |
          flutter build ipa --release \
            --export-options-plist=/tmp/ExportOptions.plist

    artifacts:
      - build/ios/ipa/*.ipa

    publishing:
      app_store_connect:
        auth: integration
        submit_to_testflight: true
ℹ
Tärkeää Muista kasvattaa pubspec.yaml-tiedoston build numberia joka julkaisun yhteydessä. App Store Connect ei hyväksy samaa build numberia kahdesti. Esimerkiksi version: 1.0.0+1 → version: 1.0.0+2.

Yleisimmät virheet ja ratkaisut

Scheme not found

Syy: Flutter 3.29+ Swift Package Manager -bugi. Ratkaisu: lisää flutter pub get ensimmäiseksi skriptiksi ennen muita vaiheita.

No matching profiles found for bundle identifier

Syy: Codemagicin ios_signing-lohko ei löydä provisioning profiilia. Ratkaisu: varmista että profiili on lisätty Codemagicin Code signing identities -osioon ja reference name täsmää täsmälleen YAML:ssa käytettyyn nimeen.

Cannot save Signing Certificates without certificate private key

Syy: fetch-signing-files-komento yrittää hakea sertifikaattia Apple Developer Portalista, mutta yksityinen avain ei ole siellä. Ratkaisu: käytä ios_signing-lohkoa environment-osiossa ja viittaa sertifikaattiin reference nimellä.

property list does not exist

Syy: $CM_EXPORT_OPTIONS_PLIST-muuttujaa ei luoda automaattisesti kun käytetään reference name -lähestymistapaa. Ratkaisu: luo ExportOptions.plist manuaalisesti skriptissä kuten yllä olevassa YAML-esimerkissä.

Apple post processing failed – missing Beta App Information

Syy: TestFlight-lähetystä varten tarvitaan yhteystiedot. Ratkaisu: täytä tiedot App Store Connectissa kohdassa TestFlight → Test Information: Feedback Email, First Name, Last Name, Phone Number.

dot-shorthands language feature

Syy: koodi käyttää uutta Dart-syntaksia jonka Codemagicin Flutter-versio ei tue. Ratkaisu: korvaa lyhytsyntaksi täydellisellä, esim. .center → MainAxisAlignment.center.

09

TestFlight-jakelu testaajille

Sisäinen testaus (Internal Testing)

Ei vaadi Applen hyväksyntää, kutsu tulee heti. Testaajien tulee olla App Store Connect -tilin käyttäjiä (Admin, Developer tms. rooli).

App Store Connect→ TestFlight→ Internal Testing→ Builds-välilehti→ Lisää build ryhmään

Ulkoinen testaus (External Testing)

Vaatii Applen beta review'n (1–2 päivää). Testaajat voivat olla keitä tahansa – heillä täytyy olla vain TestFlight asennettuna.

  1. Täytä TestFlight Test Information -yhteystiedot ennen lähetystä.
  2. Lisää testaajat External Testing -ryhmään sähköpostiosoitteella.
  3. Klikkaa Submit for Review.
  4. Odota Applen hyväksyntä (yleensä alle 2 päivää).
  5. Testaajat saavat sähköpostikutsun TestFlightiin.
✓
Vinkki Voit lisätä itsesi sekä Internal että External Testing -ryhmiin samanaikaisesti. Internal Testing -ryhmän kautta saat sovelluksen heti puhelimellesi ilman review-odotusta.
